# Phase 01.1: Native Shell Polish - Research

**Researched:** 2026-03-26
**Domain:** SwiftUI native chrome -- navigation, animations, state persistence, accessibility
**Confidence:** HIGH

## Summary

This phase replaces the existing `TabView`-based navigation shell with a custom floating pill navigation button, home screen hub, context-aware loading screens, auth half-sheet presentation, and dark background blending. All work is pure native SwiftUI/UIKit chrome -- no CSS/JS injection or web content changes.

The existing codebase is well-structured for this refactor. The critical architectural constraint is preserving the "ZStack-always-rendered" pattern where both `PlatformWebView` instances remain in the view hierarchy at all times to avoid destroying WKWebView session state. The `TabView` currently provides this implicitly (it keeps both tab views alive); the replacement must explicitly maintain both web views in a ZStack and toggle visibility via opacity. All required APIs (`DragGesture`, `@AppStorage`, `.presentationDetents`, `@Environment(\.accessibilityReduceMotion)`, `.ultraThinMaterial`) are stable, well-documented iOS 17+ SwiftUI features.

**Primary recommendation:** Structure the new `ContentView` as a single ZStack containing: (1) both platform web views always rendered at opacity 0 or 1, (2) a home screen view, (3) the floating pill overlay, with navigation state driven by a single `@Observable` navigation model using `@AppStorage` for persistence.

### Pattern 1: ZStack-Always-Rendered with Opacity Navigation
**What:** Replace TabView with a ZStack that keeps all web views alive and toggles visibility via opacity.
**When to use:** Always -- this is the critical architectural pattern that prevents WKWebView session destruction.
**Why:** TabView keeps both tab views in memory implicitly. A custom navigation must do this explicitly. If a web view is removed from the SwiftUI hierarchy, WKWebView is deallocated and the user loses their session.
**Example:**
```swift
// ContentView.swift - core navigation structure
struct ContentView: View {
    @State private var nav = NavigationState()
    @State private var instagramState = WebViewState()
    @State private var youtubeState = WebViewState()

    var body: some View {
        ZStack {
            // Layer 0: Both web views ALWAYS rendered (session preservation)
            PlatformTabView(platform: .instagram, state: instagramState)
                .opacity(nav.activePlatform == .instagram ? 1 : 0)
                .allowsHitTesting(nav.activePlatform == .instagram)

            PlatformTabView(platform: .youtube, state: youtubeState)
                .opacity(nav.activePlatform == .youtube ? 1 : 0)
                .allowsHitTesting(nav.activePlatform == .youtube)

            // Layer 1: Home screen (shown when no platform active)
            if nav.activeScreen == .home {
                HomeScreenView(onSelectPlatform: { platform in
                    nav.navigate(to: platform)
                })
                .transition(.opacity)
            }

            // Layer 2: Loading overlays (per platform)
            // Layer 3: Floating pill button (always on top)
            FloatingPillButton(nav: nav)
        }
        .animation(.easeInOut(duration: 0.25), value: nav.activeScreen)
    }
}
```

### Pattern 2: Navigation State Model
**What:** A single `@Observable` class that owns all navigation state: active screen (home/instagram/youtube), pill expanded state, pill position, and last-used platform.
**When to use:** For all navigation logic in this phase.
**Example:**
```swift
@Observable
@MainActor
class NavigationState {
    enum Screen: String {
        case home
        case instagram
        case youtube
    }

    // Persisted state
    @ObservationIgnored
    @AppStorage("lastPlatform") private var lastPlatformRaw: String = ""

    // Transient state
    var activeScreen: Screen = .home
    var isPillExpanded: Bool = false
    var pillPosition: CGPoint = .zero // loaded from UserDefaults

    var activePlatform: Platform? {
        switch activeScreen {
        case .home: return nil
        case .instagram: return .instagram
        case .youtube: return .youtube
        }
    }

    func navigate(to screen: Screen) {
        activeScreen = screen
        isPillExpanded = false
        if screen != .home {
            lastPlatformRaw = screen.rawValue
        }
    }

    func restoreLastPlatform() {
        if let screen = Screen(rawValue: lastPlatformRaw), screen != .home {
            activeScreen = screen
        }
        // else: stays on .home (first launch)
    }
}
```

### Pattern 3: Draggable Floating View with Position Persistence
**What:** `DragGesture` on the pill button that updates position in real-time and saves to UserDefaults on drag end.
**When to use:** For the floating pill button (D-04, D-07).
**Example:**
```swift
struct FloatingPillButton: View {
    @Bindable var nav: NavigationState
    @State private var position: CGPoint = Self.defaultPosition
    @State private var dragOffset: CGSize = .zero

    static let defaultPosition = CGPoint(x: UIScreen.main.bounds.width - 44, y: UIScreen.main.bounds.height - 120)

    var body: some View {
        pillContent
            .position(position)
            .offset(dragOffset)
            .gesture(
                DragGesture()
                    .onChanged { value in
                        dragOffset = value.translation
                    }
                    .onEnded { value in
                        position = CGPoint(
                            x: position.x + value.translation.width,
                            y: position.y + value.translation.height
                        )
                        dragOffset = .zero
                        // Clamp to safe area
                        clampToSafeArea()
                        // Save to UserDefaults
                        savePillPosition()
                    }
            )
            .onAppear { loadPillPosition() }
    }

    private func savePillPosition() {
        UserDefaults.standard.set(position.x, forKey: "pillPositionX")
        UserDefaults.standard.set(position.y, forKey: "pillPositionY")
    }

    private func loadPillPosition() {
        let x = UserDefaults.standard.double(forKey: "pillPositionX")
        let y = UserDefaults.standard.double(forKey: "pillPositionY")
        if x != 0 || y != 0 {
            position = CGPoint(x: x, y: y)
        }
    }
}
```

### Pattern 4: Reduce Motion Accessibility
**What:** Respect the system Reduce Motion setting by conditionally disabling animations.
**When to use:** All animations in this phase (pill expand/collapse, fade transitions, spring animations).
**Example:**
```swift
@Environment(\.accessibilityReduceMotion) private var reduceMotion

// Use conditional animation
.animation(reduceMotion ? nil : .spring(response: 0.35, dampingFraction: 0.8), value: isPillExpanded)
.animation(reduceMotion ? nil : .easeInOut(duration: 0.25), value: activeScreen)
```

### Anti-Patterns to Avoid
- **Conditional web view rendering:** NEVER use `if platform == .instagram { PlatformWebView(...) }` -- this destroys the WKWebView when the condition is false, killing the user's session. Use opacity + allowsHitTesting instead.
- **NavigationStack for platform switching:** Do not use NavigationStack/NavigationLink for switching between platforms. NavigationStack manages a stack of views and will deallocate popped views. Use a flat ZStack.
- **@AppStorage for CGPoint:** `@AppStorage` does not directly support CGPoint. Store x and y as separate Double values, or encode as a JSON string. Separate doubles is simpler and more performant.
- **GeometryReader for pill positioning:** Avoid wrapping the entire content in GeometryReader. Use `UIScreen.main.bounds` for initial default positioning and safe area insets for clamping.
- **Forgetting allowsHitTesting:** Setting opacity to 0 does NOT disable touch interaction in SwiftUI. A hidden web view at opacity 0 will still intercept taps. Always pair `.opacity(0)` with `.allowsHitTesting(false)`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Material blur background | Custom blur effect with UIVisualEffectView | `.ultraThinMaterial` modifier | Apple's material system handles dark/light mode, accessibility, and performance automatically |
| Half-sheet presentation | Custom bottom-sheet with drag gestures | `.presentationDetents([.medium])` | Built-in iOS 16+ API, handles all edge cases (keyboard avoidance, rotation, accessibility) |
| Safe area measurement | Manual calculation of safe area insets | `safeAreaInsets` environment value or `GeometryReader` | System provides correct values that update on rotation |
| Haptic feedback on pill tap | AudioServicesPlaySystemSound | `UIImpactFeedbackGenerator(style: .light)` | Standard iOS haptic API, respects system haptic settings |
| Reduce motion detection | Manual notification observer for UIAccessibility | `@Environment(\.accessibilityReduceMotion)` | SwiftUI environment automatically triggers view updates |

## Common Pitfalls

### Pitfall 1: WKWebView Destruction on View Removal
**What goes wrong:** Switching from TabView to custom navigation accidentally removes a PlatformWebView from the hierarchy, destroying the WKWebView and losing the user's session (cookies, scroll position, navigation history).
**Why it happens:** SwiftUI conditionally renders views -- an `if/else` or `switch` statement removes views from the hierarchy entirely.
**How to avoid:** Always render both PlatformWebView instances in the ZStack. Use `.opacity(0/1)` and `.allowsHitTesting(true/false)` to show/hide. Never use conditional rendering for web view containers.
**Warning signs:** User gets logged out when switching platforms. Web view reloads from scratch on platform switch.

### Pitfall 2: allowsHitTesting Not Set
**What goes wrong:** A hidden (opacity 0) web view intercepts all touch events, making the visible content unresponsive.
**Why it happens:** SwiftUI's opacity modifier does not disable hit testing. A fully transparent view still receives gestures.
**How to avoid:** Always pair `.opacity(0)` with `.allowsHitTesting(false)` and `.opacity(1)` with `.allowsHitTesting(true)`.
**Warning signs:** Taps on visible content do nothing, or the wrong web view scrolls.

### Pitfall 3: Pill Button Goes Behind Safe Area
**What goes wrong:** User drags the pill button behind the home indicator, notch, or dynamic island where it becomes unreachable.
**Why it happens:** DragGesture provides unclamped translation values.
**How to avoid:** In the `.onEnded` handler, clamp the position to stay within screen bounds minus safe area insets minus a margin (8pt per UI-SPEC). Use `UIApplication.shared.connectedScenes` to get the current window's safe area insets.
**Warning signs:** Button disappears behind system UI elements.

### Pitfall 4: @AppStorage Enum Conformance
**What goes wrong:** Using `@AppStorage` with a custom enum crashes or fails to compile.
**Why it happens:** `@AppStorage` only supports types that conform to `RawRepresentable` with a `RawValue` of `String` or `Int`.
**How to avoid:** Ensure the navigation screen enum has `String` as its raw value type. For `Platform`, which already conforms to `String, RawRepresentable`, this works directly.
**Warning signs:** Compiler error about RawRepresentable conformance.

### Pitfall 5: Animation Conflict with Multiple .animation Modifiers
**What goes wrong:** Multiple `.animation()` modifiers on the same view tree cause unexpected animation behavior -- elements animate when they should not, or use the wrong curve.
**Why it happens:** SwiftUI's `.animation(_:value:)` applies to all animatable changes in the subtree.
**How to avoid:** Use explicit `withAnimation {}` blocks for navigation changes rather than implicit `.animation()` modifiers. This gives precise control over which state changes animate and which do not.
**Warning signs:** Loading indicators animate with the wrong curve, pill button animates when it should not.

### Pitfall 6: Auth Sheet Binding Breaks During Navigation Restructure
**What goes wrong:** The `.sheet(item: $state.pendingAuthURL)` binding stops working after restructuring ContentView.
**Why it happens:** The sheet binding was on PlatformTabView. If PlatformTabView is no longer the immediate parent managing the sheet, the binding path changes.
**How to avoid:** Keep the `.sheet(item:)` on PlatformTabView or carefully migrate it to the restructured ContentView. Ensure `WebViewState.pendingAuthURL` is still accessible from wherever the sheet binding lives.
**Warning signs:** Auth redirects no longer show the sign-in modal.

### Pitfall 7: Spring Animation Parameters
**What goes wrong:** Pill expand/collapse animation feels sluggish, bouncy, or jarring.
**Why it happens:** Default spring parameters may not match the app's dark/minimal aesthetic.
**How to avoid:** Use the UI-SPEC specified values: `.spring(response: 0.35, dampingFraction: 0.8)` for pill expand/collapse. These give a snappy, slightly-damped feel without excessive bounce.
**Warning signs:** Animation feels "playful" rather than "calm."

## Open Questions

1. **Pill button z-ordering with system overlays**
   - What we know: The pill floats as a SwiftUI overlay in the ZStack, positioned above web content.
   - What's unclear: Whether iOS system overlays (keyboard, control center edge gestures) will conflict with pill positioning at screen edges.
   - Recommendation: Default position is bottom-trailing, 16pt from safe area edges. Clamping to safe area + 8pt margin should avoid conflicts. Test on simulator.

2. **Pill tap detection vs. web view scroll**
   - What we know: The pill overlays the web view. SwiftUI hit testing follows view hierarchy order (last in ZStack wins).
   - What's unclear: Whether fast scrolling web content could interfere with pill tap detection.
   - Recommendation: The pill is in a higher ZStack layer, so it should always receive taps first. Verify in simulator testing.
