# Phase 01.1 Context: Native Shell Polish

> Pre-loaded from user conversation before discuss-phase. Skip any questions that are already answered here.

---

## Why This Phase Exists

After Phase 1 shipped a functional WKWebView shell, user review identified that the app feels like "a web app rendered inside a native app" rather than a cohesive product. Phase 01.1 addresses the native shell experience before Phase 2 begins injecting CSS into the web content. Getting the native chrome right first means Phase 2 can focus purely on the injection pipeline.

---

## User Goals

- Make the app look and feel like one cohesive product — not a web app inside a native container
- Make navigation intuitive and easy
- Give social media content as much screen space as possible — it should feel native, not like a smaller window inside another app

---

## Specific UI Changes Required

These are the key areas the user explicitly wants addressed. This list is NOT exhaustive — a general dark, minimal overhaul of the native chrome is also expected.

### 1. Navigation Overhaul (DECIDED: Floating Button)

**Decision made:** Replace the bottom `TabView` tab bar with a floating expandable button.

**Why floating button over gestures (already resolved — do NOT re-open):**
- WKWebView already owns horizontal swipe gestures (back/forward browser navigation)
- Adding swipe-to-switch-tabs would conflict with WKWebView's gesture recognizers
- Floating button is discoverable, doesn't conflict, and scales to future platforms
- Movable by the user so it doesn't permanently block content

**Expected behavior:**
- Floating button sits over the web content (not part of the web view)
- User can drag/reposition it
- Tap to expand → shows navigation options (platforms, home, settings entry point)
- Collapse on tap-away or selection

### 2. Home Screen / Hub

**Decision made:** Build a ZenSocial home screen as the app's landing state.

**Why (already resolved — do NOT re-open):**
- Gives the app a proper branded identity before any web content loads
- The floating button's primary destination — tapping "home" on the floating button returns here
- Provides a natural place to launch platforms and surface future navigation (Settings stub)
- Scales cleanly as more platforms are added

**Expected content:**
- ZenSocial branding
- Platform launcher cards/buttons (Instagram, YouTube) — tapping opens the platform
- Settings entry point (stub — grayed out or navigates to empty screen; NOT implemented yet)
- Clean, dark, minimal aesthetic matching the overall app theme

**Out of scope for this phase:**
- Actual Settings screen implementation (Phase 4)
- Any backend, accounts, or sync

### 3. Custom Loading Screens

Replace the current generic `ProgressView` spinner with custom, context-aware loading screens.

**State-specific variants needed:**
- Loading Instagram
- Loading YouTube
- Waiting for Sign In (when auth is pending)
- General/fallback loading state

**Design direction:**
- Match ZenSocial's dark, minimal aesthetic
- Each should feel distinct but cohesive
- Platform-specific: use platform identity (color, name) to make it clear what's loading

### 4. Auth Modal as Half-Sheet

**Current behavior:** Auth modal slides up as a full-screen sheet.

**Desired behavior:** Appears as a half-sheet / modal sheet — smaller, leaving the page visible and dimmed behind it. Feels like a modal overlay, not a full navigation push.

**Implementation:** SwiftUI `.presentationDetents([.medium])` or similar. The web content behind should remain visible and dimmed.

### 5. Background Blending

**Goal:** The background of whatever social media platform is open should visually blend into the native iOS app chrome — so the transition between web content and native UI elements feels seamless, not like two separate layers.

**What this means technically:**
- Set a consistent dark background color on the WKWebView's native `backgroundColor` property so there's no white flash or mismatched border
- Native SwiftUI elements (safe area, any chrome around the web view) should match the web content's background
- The CSS injection side of this (forcing a known dark background on web content) is handled in Phase 2 — Phase 01.1 handles the native side (matching SwiftUI chrome to the web content background)

## Architectural Notes from Phase 1

These decisions from Phase 1 affect Phase 01.1 implementation:

- **TabView is the current navigation shell** — Phase 01.1 replaces it. The `ContentView.swift` and `PlatformTabView.swift` are the primary files to modify.
- **WebViewState is held at ContentView level** — this pattern (state survives tab switches) should be preserved whatever navigation system replaces TabView.
- **PlatformWebView is always rendered in a ZStack** (even during error state) — preserves WKWebView session. Keep this pattern.
- **Auth modal uses `.sheet(item: $state.pendingAuthURL)`** — change the sheet presentation style but keep the binding mechanism.
- **NetworkMonitor and UserAgentProvider** are injected via `.environment()` — should continue working regardless of navigation restructuring.

## Scope Boundaries

**In scope for Phase 01.1:**
- Home screen (branded hub, platform launchers, Settings stub)
- Floating expandable navigation button (replaces TabView tab bar)
- Custom context-aware loading screens
- Auth modal as half-sheet (`.presentationDetents`)
- Background blending (native side — WKWebView backgroundColor matching)
- General dark/minimal polish of native chrome

**Out of scope for Phase 01.1:**
- CSS/JS injection into web content (Phase 2)
- Feature blocking — Reels/Shorts (Phase 3)
- Settings implementation (Phase 4)
- Backend, accounts, or sync (future milestone)
- Third platform support (future)

**Ambiguous / needs discussion:**
- Exactly how the floating button expands (radial menu? vertical list? popover?)
- Whether the home screen is the default launch state, or whether app remembers last platform
- Drag-to-reposition persistence — does the button position save between sessions?
- Transition animations between home and platform views
