# Phase 01.1 Context: Native Shell Polish

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the TabView-based shell with a polished native chrome: a home screen hub, a floating pill navigation button, context-aware loading screens, an auth half-sheet, and background blending. This is native chrome only — no CSS/JS injection into web content (Phase 2).

</domain>

<decisions>
## Implementation Decisions

### Navigation — Floating Pill Button (replaces TabView)

- **D-01:** The existing `TabView` tab bar is replaced by a floating circular button that expands into a horizontal pill dock on tap.
- **D-02:** Pill contains three items: **Home · Instagram · YouTube**. Icon-based only — no labels unless long-pressed. No Settings in the pill.
- **D-03:** Settings is accessible from the home screen only — keeps the switcher lean and scales cleanly when more platforms are added.
- **D-04:** The pill button floats over web content (not part of the web view). User can drag to reposition.
- **D-05:** Collapses on tap-away or after a selection is made.

### Launch Behavior

- **D-06:** First launch → home screen. Subsequent launches → last platform used, saved via `@AppStorage`.

### Button Position Persistence

- **D-07:** Drag position is saved to UserDefaults and restored on next launch. Button does not reset to default corner.

### Transition Animations

- **D-08:** Fade transitions between home screen and platform views (and between platforms). Simple, minimal, fits the dark aesthetic.
- **D-09:** Animations are a future revisit candidate — fade is the baseline, not a final answer.

### Home Screen

- **D-10:** ZenSocial home screen is the app's landing state on first launch. Contains: ZenSocial branding, platform launcher cards (Instagram, YouTube), and a Settings entry point.
- **D-11:** Settings button on home screen is tappable → navigates to an empty placeholder screen. Not grayed out.

### Auth Modal

- **D-12:** Auth modal presented as a half-sheet using `.presentationDetents([.medium])`. Web content remains visible and dimmed behind it. The `.sheet(item: $state.pendingAuthURL)` binding mechanism is preserved — only the presentation style changes.

### Custom Loading Screens

- **D-13:** Replace `ProgressView` spinner with context-aware loading screens for: Loading Instagram, Loading YouTube, Waiting for Sign In, and General/fallback.
- **D-14:** Each variant uses platform identity (color, name) to make it clear what's loading. Dark, minimal aesthetic throughout.

### Background Blending

- **D-15:** Set a consistent dark background color on `WKWebView.backgroundColor` so there's no white flash or mismatched border between web content and native chrome.
- **D-16:** Native SwiftUI elements (safe area, any chrome around the web view) match the web content's dark background. CSS-side forcing of dark background is Phase 2.

### Claude's Discretion

- Exact drag-to-reposition UX mechanics (snap-to-edges, inertia, bounds clamping) — standard iOS drag gesture behavior is fine.
- Pill expand/collapse animation style and duration — keep consistent with fade aesthetic.
- Empty Settings placeholder screen content — a simple "Settings coming soon" label is sufficient.
- Exact loading screen layout within the dark/minimal constraint.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `WebViewState` (`@Observable`) — holds loading state, pendingAuthURL, etc. Held at ContentView level; must survive navigation restructure.
- `PlatformWebView` — the WKWebView wrapper. Always rendered in a ZStack even during error state. Keep this pattern.
- `Color.zenAccent` (#4DA6FF) — established accent color for tinting.
- `.sheet(item: $state.pendingAuthURL)` — binding mechanism for auth modal. Keep the binding, change only the presentation style.
- `NetworkMonitor`, `UserAgentProvider` — injected via `.environment()`. Will continue working regardless of navigation restructure.

### Established Patterns
- `ZStack`-always-rendered: WKWebView is always in the view hierarchy (never conditionally removed) to preserve session state.
- State at top level: `instagramState` and `youtubeState` are held on `ContentView`, not inside tab views.
- `@Observable` macro (Observation framework, iOS 17+) for app state.

### Integration Points
- `ContentView.swift` — primary file to restructure; replace TabView with floating pill + home screen navigation.
- `PlatformTabView.swift` — ZStack and loading/error overlay patterns to preserve or migrate.

</code_context>

<specifics>
## Specific Ideas

- Pill design: circular button → expands horizontally into a pill shape. The circular collapsed state is the "handle" users drag to reposition.
- Platform launchers on home screen should use platform identity (color, name/icon) — consistent with the loading screen approach.
- Fade transitions: user explicitly noted these may be revisited as the app's visual identity develops.

</specifics>

<deferred>
## Deferred Ideas

- More sophisticated transition animations (zoom, push) — user flagged fade as a baseline, not final. Revisit in a future polish pass.
- TikTok and additional platform support — pill scales to accommodate, but TikTok is explicitly out of scope for this milestone.
- Actual Settings screen implementation — Phase 4.
- Multi-account support, push notifications, backend — future milestone.

</deferred>
