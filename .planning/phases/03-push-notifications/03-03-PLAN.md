<objective>
Build the backend notification relay server and wire it to the native iOS app based on the architecture decision from Plan 02. The backend monitors Instagram notification state for registered users and sends visible APNs pushes to their devices. This satisfies PUSH-02 (background delivery) and PUSH-03 (force-quit delivery) -- both require server-sent visible APNs notifications because silent pushes do not wake force-quit apps.

Purpose: Force-quit notification delivery is what makes ZenSocial a real Instagram replacement (per D-05). Visible APNs notifications are delivered by iOS at the OS level regardless of app process state -- foreground, background, suspended, or terminated.

Output: Backend relay server, APNs token registration endpoint, Instagram notification monitoring, updated APNsTokenManager to register with backend, human-verified end-to-end notification flow.

**CRITICAL CONSTRAINT FROM RESEARCH:** Web Push does NOT work in WKWebView. DO NOT use PushManager.subscribe() or any web push JavaScript injection -- this will fail silently in WKWebView per RESEARCH.md. All notification delivery goes through native APNs only. The backend independently monitors Instagram notifications and sends APNs pushes.
</objective>

    **If option-a (cookie-based polling) or option-b (FBNS):**

    1. Create `Backend/` directory at project root with a lightweight server (Vapor/Hummingbird in Swift, or Express in Node.js -- Claude's discretion per research).
    2. Backend endpoints:
       - `POST /register` -- accepts `{ deviceToken: string, deviceId: string }`, stores mapping
       - `POST /unregister` -- removes device from monitoring
    3. Backend APNs sending: Use APNSwift (Swift) or `apn` npm package (Node.js) to send visible pushes with the payload format from interfaces section. Must use JWT-based auth with the p8 key from Apple Developer Portal.
    4. Backend Instagram monitoring: Implement the chosen approach (polling endpoint or FBNS connection). Parse notification data to extract: notification type, sender name, notification text, deep-link URL.
    5. Update `APNsTokenManager.swift`: Add `func registerWithBackend() async` that POSTs the device token to the backend's `/register` endpoint. Call this after successful token registration in AppDelegate's `didRegisterForRemoteNotificationsWithDeviceToken`.

    **If option-c (on-device background fetch):**

    1. No backend directory needed.
    2. Register a `BGAppRefreshTask` in `ZenSocialApp.swift` using `BGTaskScheduler`.
    3. In the background task handler, use the existing Instagram WKWebView to check notification state (badge count, notification indicator) via JavaScript evaluation.
    4. If new notifications detected, create a `UNNotificationRequest` (local notification) with the notification content.
    5. Document the limitation: notifications do not arrive after force-quit.
    6. `APNsTokenManager.swift` changes are minimal -- no backend registration needed.

    **Common to all options:**
    - Ensure push payload includes `instagram_url` key for deep-linking (matches NotificationManager.handleNotificationTap from Plan 01).
    - Ensure push payload includes `aps.alert.title` and `aps.alert.body` for visible notification.
    - Test with a test push (use `apns-tool` or similar CLI to send a test push to the device token).
