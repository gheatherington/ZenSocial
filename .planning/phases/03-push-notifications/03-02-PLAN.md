  <action>
    Present the user with three options for how the ZenSocial backend should monitor Instagram notifications.

    Research confirms Web Push does not work in WKWebView. The backend must independently detect new Instagram notifications and send visible APNs pushes. Three approaches exist, each with significant tradeoffs around privacy, ToS compliance, reliability, and complexity.

    **The core challenge:** Instagram provides no public API for subscribing to personal account notifications. All approaches involve some degree of unofficial access.

    **Option A -- Cookie-based polling:**
    Pros: Simplest to implement (backend makes authenticated HTTP requests to Instagram notification endpoint using user session cookies). No reverse-engineered protocols. Familiar web scraping pattern.
    Cons: User must transfer Instagram cookies to ZenSocial backend (privacy). Cookies expire. Instagram blocks automated requests (account ban risk). Violates Meta ToS. Polling latency (30-60s delay).

    **Option B -- FBNS (Instagram MQTT protocol):**
    Pros: Real-time delivery (no polling delay). Same protocol as native Instagram app. Community libraries exist (instagram_mqtt).
    Cons: Reverse-engineered private protocol; breaks without warning. Requires Instagram credentials on backend. Same ToS/privacy concerns as A. Higher complexity (MQTT management). Low library maintenance.

    **Option C -- Defer backend, on-device background fetch only:**
    Pros: No backend server (no infra cost, no ToS risk, no privacy concerns). Uses BGTaskScheduler for periodic checks while app is in memory. Simplest architecture; credentials never leave device.
    Cons: Does NOT satisfy D-05 (force-quit delivery). Background fetch timing unreliable (15 min to hours). Requires relaxing force-quit hard requirement to documented limitation.

    Wait for user selection before proceeding.
