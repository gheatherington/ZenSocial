<objective>
Build the complete native iOS notification infrastructure for Instagram push notifications. This includes the AppDelegate bridge for SwiftUI, UNUserNotificationCenter setup, APNs device token registration, permission flow (pre-prompt + system dialog), foreground notification handling (native banner), notification tap deep-linking to Instagram content, entitlements configuration, Info.plist background modes, a Settings toggle for notifications, and updated test stubs from Wave 0.

Purpose: Establishes all native-side plumbing so that when the backend relay (Plan 02) starts sending APNs pushes, the device is ready to receive, display, and route them. Also satisfies PUSH-01 (foreground display) end-to-end once pushes arrive.

Output: NotificationManager service, APNsTokenManager service, AppDelegate with push callbacks, NotificationPermissionView, updated SettingsPlaceholderView with notification toggle, entitlements file, Info.plist background modes, updated REQUIREMENTS.md, green test stubs.
</objective>

    **1. Create `ZenSocial/Services/NotificationManager.swift`:**
    - `@Observable @MainActor class NotificationManager` (singleton via `static let shared`)
    - Properties: `var permissionStatus: UNAuthorizationStatus = .notDetermined`, `var showPrePrompt: Bool = false`, `var hasRequestedPermission: Bool` (backed by `@AppStorage("hasRequestedNotificationPermission")`)
    - Method `checkPermissionStatus()` — calls `UNUserNotificationCenter.current().notificationSettings()` and updates `permissionStatus`
    - Method `requestPermission() async -> Bool` — per D-09, this is called AFTER the user taps "Enable" on the pre-prompt. Calls `UNUserNotificationCenter.current().requestAuthorization(options: [.alert, .sound, .badge])`. If granted, calls `await MainActor.run { UIApplication.shared.registerForRemoteNotifications() }`. Sets `hasRequestedPermission = true`. Returns grant result.
    - Method `triggerPermissionFlowIfNeeded()` — per D-07, checks: (a) `hasRequestedPermission == false`, (b) `permissionStatus == .notDetermined`. If both true, sets `showPrePrompt = true`.
    - Method `handleNotificationTap(userInfo: [AnyHashable: Any])` — per D-11, extracts `instagram_url` key from userInfo as String, converts to URL. If valid, calls `NavigationState.shared.navigateToInstagram(url:)`.

    **2. Create `ZenSocial/Services/APNsTokenManager.swift`:**
    - `@MainActor class APNsTokenManager` (singleton via `static let shared`)
    - Property: `var deviceToken: String?` (stored in `@AppStorage("apnsDeviceToken")`)
    - Method `registerToken(_ tokenData: Data)` — converts Data to hex string via `tokenData.map { String(format: "%02.2hhx", $0) }.joined()`, stores in `deviceToken`. Logs to console: `"APNs token registered: \(token.prefix(8))..."`.
    - Method `getToken() -> String?` — returns stored token.

    **3. Modify `ZenSocial/Models/NavigationState.swift`:**
    - Change from local `@State` instantiation to `static let shared = NavigationState()` singleton pattern. This is necessary because notification tap handling (from AppDelegate) needs to route navigation without having access to ContentView's local state.
    - Add method `func navigateToInstagram(url: URL)` that sets `activeScreen = .instagram` and posts a `Notification.Name.zenSocialNavigateToURL` notification with the URL in userInfo `["url": url]`. The WebViewCoordinator for Instagram will observe this and call `webView.load(URLRequest(url:))`.
    - Add `static let zenSocialNavigateToURL = Notification.Name("zenSocialNavigateToURL")` in a Notification.Name extension (can add to the existing extension in PlatformTabView.swift or create inline).

    **4. Update `ZenSocial/Views/ContentView.swift`:**
    - Change `@State private var nav = NavigationState()` to use the shared singleton: `private var nav = NavigationState.shared`.

    **5. Create AppDelegate class in `ZenSocial/ZenSocialApp.swift`:**
    - Add `class AppDelegate: NSObject, UIApplicationDelegate, UNUserNotificationCenterDelegate` above the `ZenSocialApp` struct.
    - `func application(_:didFinishLaunchingWithOptions:)` — sets `UNUserNotificationCenter.current().delegate = self`, returns true.
    - `func application(_:didRegisterForRemoteNotificationsWithDeviceToken:)` — calls `APNsTokenManager.shared.registerToken(deviceToken)`.
    - `func application(_:didFailToRegisterForRemoteNotificationsWithError:)` — prints error.
    - `func userNotificationCenter(_:willPresent:) async -> UNNotificationPresentationOptions` — per D-10, returns `[.banner, .sound, .badge]` for foreground display.
    - `func userNotificationCenter(_:didReceive:) async` — per D-11, calls `NotificationManager.shared.handleNotificationTap(userInfo: response.notification.request.content.userInfo)`.
    - Add `@UIApplicationDelegateAdaptor(AppDelegate.self) var appDelegate` to `ZenSocialApp`.
    - Mark AppDelegate methods with `@MainActor` where needed for Swift 6 strict concurrency.

    **6. Create `ZenSocial/ZenSocial.entitlements`:**
    ```xml
    <?xml version="1.0" encoding="UTF-8"?>
    <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
    <plist version="1.0">
    <dict>
        <key>aps-environment</key>
        <string>development</string>
    </dict>
    </plist>
    ```
    - Add this entitlements file to the Xcode project's build settings (CODE_SIGN_ENTITLEMENTS = ZenSocial/ZenSocial.entitlements). This requires updating the pbxproj or doing it via Xcode build settings in the project file.

    **7. Add UIBackgroundModes: remote-notification to the Xcode build settings:**
    - The project uses `GENERATE_INFOPLIST_FILE = YES` (no custom Info.plist). To add UIBackgroundModes, add the following build setting to the main target's build settings in project.pbxproj:
      `INFOPLIST_KEY_UIBackgroundModes = "remote-notification";`
    - This tells Xcode to inject `<key>UIBackgroundModes</key><array><string>remote-notification</string></array>` into the generated Info.plist at build time.
    - This is REQUIRED for APNs background delivery (PUSH-02). Without it, iOS will not wake the app for silent pushes.

    **8. Update WebViewCoordinator to detect Instagram login and trigger permission flow:**
    - In `WebViewCoordinator.swift`, in the `webView(_:didFinish:)` or URL observation, detect when the URL transitions from an Instagram auth domain (accounts.google.com, login.instagram.com) to the Instagram feed (instagram.com with path "/" or "/feed" or similar non-auth path). When detected, call `NotificationManager.shared.triggerPermissionFlowIfNeeded()`.
    - Also add observer for `zenSocialNavigateToURL` notification: when received and `platform == .instagram`, extract URL from userInfo and call `webView?.load(URLRequest(url: url))`.

    **Important concurrency notes:**
    - AppDelegate must use `@MainActor` annotation or use `nonisolated` with explicit MainActor dispatching for delegate methods per Swift 6 strict concurrency.
    - The `async` variants of `UNUserNotificationCenterDelegate` methods (`willPresent` and `didReceive`) handle concurrency correctly.
