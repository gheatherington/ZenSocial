# Phase 01.2 Context: Fixes for Phase 1 and 1.1 Features

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Targeted bug fixes and polish for the features built in Phase 1 (WKWebView shell) and Phase 1.1 (floating pill navigation, home screen, loading screens). No new capabilities. Scope is strictly the five issues listed in the Fixes section of gTodo.md:

1. Stop playing videos after platform has been closed (YouTube audio continuing on switch)
2. Add movement animation when dragging the pill button
3. Prevent expanded pill from going off-screen when near an edge
4. Make platform content extend to the bottom of the screen (edge-to-edge)
5. Hide the FloatingPillButton when not on a social media platform

</domain>

<decisions>
## Implementation Decisions

### Video Playback on Platform Switch

- **D-01:** When a platform becomes inactive (user switches to another platform or to Home), inject `document.querySelectorAll('video').forEach(v => v.pause())` into that platform's WKWebView. Playback position is preserved — user can resume from where they left off when returning.
- **D-02:** "Inactive" means `nav.activePlatform` is no longer this platform. Trigger in `ContentView` or `PlatformTabView` on the opacity/visibility change, not on app backgrounding (that is a separate concern).

### Floating Pill Visibility

- **D-03:** `FloatingPillButton` is only rendered when `nav.activeScreen == .instagram || nav.activeScreen == .youtube`. Hidden on home screen, settings screen, and any future non-platform screen.
- **D-04:** No fade-out — the pill is removed from the view hierarchy entirely when not on a platform screen. Clean, not just transparent.

### Pill Drag Animation

- **D-05:** Movement during drag is 1:1 with the finger — no animation applied while dragging (avoids feeling laggy or disconnected).
- **D-06:** On drag release, apply a spring animation to settle the pill to its clamped final position. Eliminates the current "teleport" snap. Use a spring consistent with existing `spring(response: 0.35, dampingFraction: 0.8)` used elsewhere in the pill.

### Expanded Pill Overflow Fix

- **D-07:** When the pill expands, detect which side of the screen it is on and expand toward the center. Near the right edge → pill expands leftward. Near the left edge → pill expands rightward. Feels like a context menu — intentional and natural.
- **D-08:** "Near an edge" threshold: if the pill's center x position is in the right half of the screen, expand left; if in the left half, expand right. Simple and consistent.

### Platform Edge-to-Edge Layout

- **D-09:** Platform web content (WKWebView) must extend to the bottom of the screen — no gap between web content and the bottom edge. This is a safe area / layout constraint bug. Fix by ensuring `PlatformWebView` / `PlatformTabView` ignores safe area at the bottom (`.ignoresSafeArea(.all, edges: .bottom)`) so the web content fills the full screen height.
- Claude's discretion on the exact modifier placement (whether on `PlatformWebView`, `PlatformTabView`, or both).

### Claude's Discretion

- Exact spring parameters for drag release animation — match existing pill spring style.
- Exact safe area modifier placement for edge-to-edge fix.
- Whether video pause injection uses `WKUserScript` or `evaluateJavaScript` (evaluateJavaScript at the moment of switch is appropriate here — not a persistent script).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `NavigationState.activePlatform` — computed property that returns `nil` on home screen. Use this as the pill visibility gate (D-03).
- `togglePill(expanded:)` — existing animation helper in `FloatingPillButton`. Spring params already defined here; match for drag-release animation.
- `clampedPosition(_:)` — existing safe-area-aware bounds clamping in `FloatingPillButton`. The overflow fix extends this to account for expanded pill width.
- `WebViewCoordinator.evaluateJavaScript` — available via `WKWebView`; appropriate for one-shot pause injection (D-01).

### Established Patterns
- ZStack-always-rendered: Both `PlatformTabView` instances are always in the hierarchy. Video pause must be triggered by the visibility change (`opacity`/`allowsHitTesting` toggle), not by view removal.
- `withAnimation(.easeInOut(duration: 0.25))` — fade transition baseline for platform switching (Phase 01.1 D-08).
- `.spring(response: 0.35, dampingFraction: 0.8)` — spring animation already used for pill expand/collapse.

### Integration Points
- Video pause: `ContentView` is the right place — it owns `nav.activePlatform` and both `WebViewState` objects. On `activePlatform` change, call `evaluateJavaScript` on the departing platform's WKWebView.
- Pill visibility: `ContentView` body — wrap `FloatingPillButton` in a conditional on `nav.activePlatform != nil`.
- Expanded overflow: `FloatingPillButton.expandedPill` and `clampedPosition` — needs expanded pill width to calculate whether it would clip.
- Edge-to-edge: `PlatformTabView` or its parent in `ContentView` — `.ignoresSafeArea` on the platform layer.

</code_context>

<specifics>
## Specific Ideas

- Video pause: user wants "pause after leaving the app [platform]" — position preserved, not stopped/reset.
- Pill visibility: user explicitly included Settings page and all non-platform screens ("whenever not on a social media render page such as Instagram and YouTube").
- Drag animation: user's pain point is the "teleporting" snap on release — spring settle on release is the fix.
- Expanded overflow: "flip direction" was user's instinct — expand toward center of screen depending on which side the pill is on.

</specifics>
